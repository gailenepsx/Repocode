In the current simulation process, transaction input files (Payments and Postings) are received and loaded into the simulation tool. � As a first step, the tool serializes these files to produce structured parquet datasets for Payments, Postings, and lookback sums, the latter containing the cumulative sum of incoming and outgoing transaction amounts. � The hit detection engine is then executed on a daily basis, using the Trans_Value_Date column as the reference date for the different configured models during Step 3. �
Following hit detection, an ALERT_ID is generated for each daily occurrence that meets the rules and specificities of the relevant model. � The resulting alerts are then consolidated into alert groups for score calculation, depending on the applicable business logic, such as individual, cluster, or user-level scoring. � At this stage, the tool applies aggregation rules such as first, list, and max, and calculates an initial score through get_alert_score(...), meaning that the alert content used for scoring is already structured according to the current ALERT_ID granularity. �
The process then calls agg_model_score(...), where alerts are regrouped by account, rule, and value date, the highest score is retained, ALERT_ID values are collected as lists, and a recurrence component is added through recurrence_score(...) before a second consolidation is performed at party, rule, and value date level. � Once all model outputs have been concatenated, the simulation tool performs a final consolidation by party and value date, sums the scores, flattens the underlying ALERT_ID lineage, filters alerts above the defined threshold, assigns a GLOBAL_ALERT_ID, and finally uses consolidate_alerts(...) to merge this mapping back into the alert population and generate the output parquet files. �
As a result, the current business process is based on a daily alert identity structure in which ALERT_ID is not only a technical identifier, but also a key element for grouping, scoring context, lineage tracking, and final alert consolidation within the simulation flow.
